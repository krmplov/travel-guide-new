#  Travel Guide
>путеводитель по городу: регистрация, списки мест, карточки мест, построение маршрута?-маршрут до места

### auth
- пользователь вводит логин и пароль
- нажимает войти
- видит загрузку
- либо ошибка, либо список мест

1. input - экран открывается без параметров

2. output
- `didLogin` - успешная авторизация

3. состояния
- форма
- loading
- error(message)

4. сценарии
    - открыли -> форма
    - нажали login -> loading
    - успешно -> output.didLogin
    - провал -> error

### place list
- показывает список мест
- по нажатию на место открывает деталку

1. input - экран открывается после авторизации

2. output
- `didSelectPlace(placeId)` - переход на выбранное место
- `didLogout` - выход из аккаунта

3. состояния
- `loading` - загрузка списка
- `content(items: [PlaceListItem])` - показ списка мест
- `error(message)`

4. сценарии
    1. открыли экран
    - View говорит VM -> `viewDidLiad`
    - VM ставит `state = .loading`
    - VM запрашивает места у репозитория
    - `state = .content(items)`
    2. выбор места
    - пользователь нажимает на ячейку
    - View сообщает VM -> `didSelect(placeId)`
    - VM отдаёт наружу `output.didSelectPlace(placeId)`
    - координатор открывает карточку

### place details
- показывает описание места, адрес, фото
- добавление в избранное
- кнопка открыть на карте

1. input
экран открывается с id места

2. output
- didClose
- didToggleFavorite(placeId: String, isFavorite: Bool)

3. состояния
- loading
- content(details: PlaceDetailsViewData) - показываем карточку
- error(message)

4. сценарии
    1. открытие экрана
    - координатор создает модуль с `placeId`
    - View говорит VM `viewDidLoad`
    - VM ставит `state = .loading`
    - VM запрашивает место у репозитория
    - `state = .content(details: PlaceDetailsViewData)`
    2. добавить или убрать из избранного
    - пользователь нажимает кнопку _избранное_
    - VM вызывает `favoritesRepository.toggle(placeId)`
    - VM обновляет `content`
    
    

## архитектура - MVVM
MVVM выбран для этого проекта, потому что приложение состоит из нескольких экранов с понятными UI-состояниями и сценариями. 
Для таких экранов удобно отделить отображение интерфейса от логики обработки действий пользователя и загрузки данных.
Такой подход хорошо подходит для тестирования: ViewModel можно проверять отдельно от интерфейса, подменяя Repository и Service моками

## Лабораторная 5

- Для экрана списка выбран `UITableView`.
- Этот вариант подходит для линейного списка мест


- `loading`: откройте экран списка после успешного логина. Во время первой загрузки показывается индикатор загрузки
- `error`: состояние появится, если список не удалось загрузить
- `empty`: состояние показывается, если сервис вернет пустой список мест
- `content`: при успешной загрузке отображается таблица со списком мест


- При нажатии на ячейку вызывается `viewModel.didSelectPlace(id:)`
- Навигатор `PlacesListNavigator` открывает экран деталей.
- Экран деталей в рамках сделан как заглушка
