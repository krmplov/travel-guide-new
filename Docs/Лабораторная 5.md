# **Лабораторная №5 — Экран списка (UIKit) + переиспользуемые компоненты**


### **Задача**


Реализовать экран со **списком фич/услуг/элементов**, который открывается после авторизации. (UITableView / UICollectionView / аналоги)


Источник данных — логика из **Лабы 4** (загрузка + формирование CellViewModel/State).

На экране должны отображаться элементы списка, по нажатию — переход на экран “деталей фичи” (реализация деталей может быть заглушкой, но навигация должна быть).

  
### **Ключевые моменты**

- UI сделать на **UIKit**.
    
- Никакой бизнес-логики (сеть/парсинг/маппинг) в UIViewController.
    
- Переиспользование: аккуратная работа с reuseIdentifier, регистрацией ячеек, конфигурацией.
    
- Экран должен уметь показывать состояния: loading / content / error / empty (или эквивалент).


### **Допустимые варианты реализации списка**

Можно использовать:

- UITableView
    
- UICollectionView
    
- другое решение для оптимизированного отображения списков (например, ручной reuse/виртуализация), если оно:
    
    - не лагирует на 100+ элементах
        
    - не приводит к неконтролируемому росту памяти
        
    - и архитектурно отделено от экрана

## **Требования (минимум)**


### **1) UI списка**

- Должен отображаться список элементов.
    
- Элементы конфигурируются от CellViewModel (не от DTO).
    
- Должна быть корректная работа с переиспользованием (в выбранном подходе):
    
    - регистрация/пул/reuseIdentifier (или эквивалент)
        
    - корректный сброс состояния при переиспользовании (особенно если есть картинки/асинхронщина)
        
    
  

Пример модели:

```swift
struct FeatureCellViewModel: Equatable {
    let title: String
    let subtitle: String?
    let rightText: String?   // например price/status
    let imageURL: URL?
}
```


### **2) Менеджер списка (аналог TableManager)**

  

Нужно вынести работу со списком в отдельную сущность (например ListManager), чтобы UIViewController не превращался в комбайн.

  
ListManager должен:

- хранить текущие элементы (CellViewModel)
    
- уметь обновлять данные
    
- обслуживать datasource/delegate (или эквивалент в вашем решении)
    
- сообщать о выборе элемента наружу через делегат/коллбек
      


Пример (идея, можно иначе):

```swift
protocol FeaturesListManagerDelegate: AnyObject {
    func didSelect(id: String)
}

final class FeaturesListManager: NSObject {
    weak var delegate: FeaturesListManagerDelegate?
    private var items: [FeatureCellViewModel] = []

    func setItems(_ items: [FeatureCellViewModel]) {
        self.items = items
    }
}

extension: FeaturesListManager: ListDataSource {...}
```


### **3) Состояния экрана**


Экран должен отображать состояния:

- loading (индикатор/заглушка)
    
- content(items) (список)
    
- empty (лейбл “пусто”)
    
- error(message) (сообщение + возможность повторить)
    


UI реализации может быть простой, важна корректность переходов между состояниями.
  

### **4) Архитектура**

- UIViewController отвечает только за отображение и пользовательские события.
    
- Загрузка/формирование данных идёт через слой презентации/домена (как вы выбрали в Лабе 2).
    
- Навигация — через router/координатор/аналог (как спроектировали).
    

  
### **5) Минимальная производительность**

  

Список должен адекватно работать на объёме **100+ элементов**:

- скролл без заметных лагов
    
- нет утечек/роста памяти от постоянных обновлений
    
- нет “пересоздания всего UI” при каждом обновлении данных (в разумных пределах)

## **Дополнительные задания (по выбору)**


### **D1. Pull-to-refresh**

Добавить обновление списка жестом (например UIRefreshControl или аналог для выбранного подхода).

### **D2. Поиск по списку**

Добавить поиск (UISearchBar) и фильтрацию элементов на уровне CellViewModel (без перезапроса сети).

### **D3. Картинки + кеширование + корректный reuse**

Если у вас есть imageURL:

- асинхронная загрузка изображений
    
- кеширование (например NSCache)
    
- корректный сброс/отмена при переиспользовании (например в prepareForReuse)
    

### **D4. Diffable Data Source**

Использовать UITableViewDiffableDataSource или UICollectionViewDiffableDataSource.

### **D5. Prefetch**

Использовать UITableViewDataSourcePrefetching / prefetch для коллекции (или аналог) для картинок/догрузки.

### **D6. Пагинация в UI**

Если в Лабе 4 есть пагинация — довести до UI: догрузка при достижении конца списка + защита от повторных запросов.
