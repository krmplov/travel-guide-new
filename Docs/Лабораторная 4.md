# **Лабораторная №4 — Сеть и формирование ViewModel для списка**

### **Задача**

  
Реализовать загрузку из сети данных для экрана со списком (экран после авторизации из предыдущей лабораторной).

UI реализовывать **не нужно** — нужно реализовать логику загрузки, парсинг и формирование моделей, готовых к отображению во view слое (ViewModel/State). Сам UI будет в 5 лабе.

  
### **Ключевые моменты**

- Загрузка не должна блокировать UI. 
    
- Полученные данные нужно парсить, например, с помощью Codable. 
    
- В view слой передаются готовые модели для отображения (минимум лишней информации, никакой логики во view слое). 
    
- API можно выбрать любое публичное, подходящее по смыслу приложению (примеры из описания лабы). 
    

---

## **Требования (минимум)**

  
### **1) Сетевой запрос**

- Данные должны загружаться из API, которое возвращает **массив** элементов (или объект, внутри которого есть массив — тогда явно указать, где именно массив).
    
- Загрузка должна происходить **асинхронно** и не блокировать текущее отображение. 
	
- Должна быть обработка ошибок:
    
    - ошибка сети/таймаута
        
    - ошибка статуса (не 2xx)
        
    - ошибка декодинга
        


### **2) Парсинг**

- DTO-модели под JSON должны быть Codable.
    
- JSONDecoder настраивать осмысленно (например keyDecodingStrategy, если надо).
    
- Использовать “динамический JSON” (типа SwiftyJSON) или что-то подобное можно только со значительной аргументацией.
    


### **3) Маппинг в модели для UI**

- DTO не должны попадать во view слой.
    
- Должен быть маппинг DTO → (опционально доменная модель) → CellViewModel.
    

  

Пример:

```swift
struct FeatureCellViewModel: Equatable {
    let id: String
    let title: String
    let subtitle: String?
    let rightText: String?
    let imageURL: URL?
}
```


### **4) Состояния загрузки**

Нужно описать состояние списка (чтобы UI мог это отобразить в Лабе 5):

- idle (опционально)
    
- loading
    
- content([CellViewModel])
    
- empty
    
- error(message)
    

  
Пример:

```swift
enum FeaturesState: Equatable {
    case idle
    case loading
    case content([FeatureCellViewModel])
    case empty
    case error(String)
}
```


### **4) Архитектура**


Решение должно быть встроено в архитектуру из Лабы 2:

- Сетевой клиент/репозиторий/сервис должны быть вынесены из UI слоя.
	
- Зависимости выражены через протоколы (чтобы можно было подменить моками).
    
- формирование состояния/вьюмоделей — в presenter/viewmodel/interactor (как у вас принято)
    
- VC/UIView слой (если есть заглушка экрана) не содержит бизнес-логики

## **Дополнительно (по выбору)**


### **D1. Нормальная модель ошибок**

Сделать свой тип ошибки (например NetworkError) и маппить её в текст/состояние для UI (не “URLError.description” напрямую).

### **D2. Отмена запроса (cancellation)**

Если повторно вызывается загрузка (например “Retry”), предыдущий запрос должен отменяться (или игнорироваться результат старого).

### **D3. Локальный fallback для отладки**

Добавить возможность загрузить данные из локального JSON-файла (например через флаг конфигурации). Это помогает, если API временно недоступно.

### **D4. Пагинация**

Реализовать постраничную загрузку (контракт + логика, UI доведём в Лабе 5).

### **D5. Кэширование**

Кэширование результата (в памяти или в файл) с понятной политикой (TTL/версия/ручной сброс).

### **Подсказка (пример базового сетевого клиента на async/await)**

```swift
protocol NetworkClient {
    func get<T: Decodable>(_ url: URL) async throws -> T
}

final class URLSessionNetworkClient: NetworkClient {
    func get<T: Decodable>(_ url: URL) async throws -> T {
        let (data, response) = try await URLSession.shared.data(from: url)
        guard let http = response as? HTTPURLResponse, 200..<300 ~= http.statusCode else {
            throw URLError(.badServerResponse)
        }
        return try JSONDecoder().decode(T.self, from: data)
    }
}
```
