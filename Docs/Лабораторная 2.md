# **Лабораторная №2 — Проектирование приложения и контрактов**


### **Задача**
  

Спроектировать приложение, выбрав архитектуру (аргументированно), описав **контракты взаимодействия**, **структуры данных** и **workflow**. Реализация UI и логики не требуется — только проектирование, протоколы и модели. 

  

Приложение должно содержать минимум 3 экрана:

- экран авторизации
    
- экран со списком фичей/услуг/возможностей
    
- экран одной фичи (детальный экран) 
    

  

### **Ключевые моменты**

- UI ориентируем на UIKit (SwiftUI не требуется). 
    
- Архитектура на выбор: MVC / MVP / MVVM / VIPER / YARCH (Clean Swift) / другая (с обоснованием). 
    
- Архитектура должна быть тестируемой: по интерфейсам должно быть понятно, как подменять зависимости моками. 
    

  

Тематика — любая из списка или своя (с согласованием). 

- Онлайн-банк / финансы (счета, карты, переводы)
    
- Крипто-приложение (портфель, курсы, деталка монеты)
    
- Социальная сеть (лента, профиль, пост)
    
- Доставка еды / маркет (категории, карточка ресторана/товара, заказ)
    
- Онлайн-справочник (каталог, карточка места/услуги)
    
- Трекер привычек / задач (список привычек, статистика/деталка, напоминания)
    
- Фитнес-приложение (планы тренировок, упражнение, прогресс)
    
- Бронирование (отели/столы/ивенты) — список вариантов, карточка, бронь
    
- Обучающее приложение (курсы, уроки, прогресс)
    
- Музыкальный/подкаст сервис (каталог, плейлист, эпизод/трек)
    
- Путеводитель по городу (места, маршрут, детальная карточка)
    
- Приложение для учета личных расходов (категории, операции, аналитика)
	
- Что-то другое

## **Требования (минимум)**

  

### **1) “Скелет” модулей (что именно вы проектируете)**

  

Нужно спроектировать **три модуля** (по экрану), и для каждого модуля описать:

- **Вход**: какие данные нужны, чтобы открыть экран
    
- **Выход**: какие события экран может отдавать наружу
    
- **Состояния UI**: (например initial/loading/content/error , но можно другим способом, должно быть понятно)
    
- **Сценарии**: 2–4 основных сценария на экран (пошагово)
    

  

### **2) Контракты (протоколы) между слоями**

  

Должны быть определены контракты:

- View ↔ Presenter/ViewModel/итд (как UI сообщает события и получает состояние)
    
- Presentation ↔ Domain (use case / interactor / service)
    
- Domain ↔ Data (репозитории/сервисы)
    
- Router/Navigator (навигация, открытие экранов)
    

  

Важно: **в контрактах не должно быть UIKit-типов**, кроме View-уровня (в домене/данных UIKit быть не должно).

  

### **3) Модели данных**

  

Нужно реализовать структуры данных, которые реально используются в контрактах, например:

- LoginRequest / LoginResponse
    
- FeatureListItem (для списка)
    
- FeatureDetails (для детального экрана)
    
- ViewState/ViewModel для каждого экрана
    
- доменные модели и ошибки (если применимо)

## **Что сдаём (обязательно)**

1. Репозиторий с кодом
    
2. В коде должны быть:
    
    - папки/файлы с контрактами и моделями по модулям
        
    - (опционально) “пустые” реализации классов (без логики), чтобы было видно, как всё собирается
        
    
3. README.md, где есть:
    
    - выбранная архитектура и короткое обоснование (5–10 строк)
        
    - список модулей и их ответственности
        
    - по каждому экрану: вход/выход/сценарии (пунктами)
        
    - список ключевых протоколов и моделей

## **Шаблон протоколов (пример, можно иначе)**

```swift
// MARK: - Auth

protocol AuthView: AnyObject {
    func render(_ state: AuthViewState)
}

protocol AuthPresenter {
    func didLoad()
    func didTapLogin(email: String, password: String)
}

protocol AuthRouter {
    func openFeaturesList()
}

struct AuthViewState: Equatable {
    var isLoading: Bool
    var errorText: String?
}

// MARK: - Data/Domain

protocol AuthService {
    func login(email: String, password: String) async throws -> UserSession
}

struct UserSession: Equatable {
    let token: String
    let userId: String
}
```

- Добавить диаграмму (простую): зависимости модулей и направление зависимостей.
